# Hello Solana

The smallest possible Solana program. It takes an instruction with no
accounts and no data, logs "Hello, Solana!" and the program's own id,
and returns success.

Not deployed; not interesting in isolation. The point is to see the
same behaviour expressed four different ways so you can pick a
framework for your own projects.

## 1. What does this program do?

Exactly one thing: it handles a single instruction with zero accounts,
zero instruction data, writes two lines to the program log, and
returns `Ok`.

The log lines are:

```
Hello, Solana!
<program id as base58>
```

There is no state. There are no PDAs. Nothing is written to chain
except the transaction's log — which is itself transient (it's stored
in transaction metadata, not in an on-chain account).

## 2. Glossary

**Program**
: A deployed, executable account on Solana. Programs are referred to
by their *program id* — a 32-byte public key — and respond to
instructions sent in transactions.

**Instruction**
: The unit of work a program processes. Each instruction names a
program id, a list of accounts, and a blob of instruction data. The
program's entrypoint receives all three.

**Transaction**
: A batch of one or more instructions signed by one or more keypairs
and submitted to a validator. Instructions within a single
transaction execute sequentially; if any one fails, the whole
transaction reverts.

**Entrypoint**
: The function the Solana runtime calls for every instruction this
program sees. In native Rust it's a free function registered via
`entrypoint!(...)`. In Anchor and Pinocchio it's generated by a
macro around your code.

**Account**
: A 32-byte-addressed chunk of state — lamports, owner program, data
bytes. "No accounts" here means this instruction doesn't touch any;
the transaction still has at least a fee-paying signer account
(that's a transaction-level concern, not an instruction input).

**Log (program log)**
: A list of strings the program writes with `msg!` (native / Anchor)
or `log!` (Pinocchio). Logs are attached to transaction metadata and
visible from RPC / `solana logs`. They are not stored in accounts;
they do not cost extra rent. Large or frequent logs still cost
compute units.

## 3. Accounts and PDAs

None. This program has no state accounts and uses no PDAs.

## 4. Instruction lifecycle walkthrough

There is exactly one instruction. Call it `hello`.

**Who calls it:** anyone with a funded wallet to pay transaction fees.

**Signers:** the fee-payer (required by every Solana transaction; not
used inside the instruction).

**Accounts in:** none required.

**Instruction data:** the program doesn't inspect it. An empty byte
string is conventional. The Anchor and Quasar variants use an 8-byte
Anchor-style discriminator as the first bytes; native and Pinocchio
don't require anything.

**Behaviour:**

1. Log "Hello, Solana!".
2. Log the program's own id.
3. Return `Ok(())` / `ProgramResult::success()`.

**Token movements:** none.

**State changes:** none.

**Checks:** none.

## 5. Worked example

After building and deploying, send a transaction with one instruction
targeting the program id, no accounts, no data. Fetch the transaction
log from the explorer or `solana confirm -v <signature>` and you
should see:

```
Program <program_id> invoke [1]
Program log: Hello, Solana!
Program log: <program_id repeated>
Program <program_id> success
```

That's the whole contract.

## 6. Safety and edge cases

None to speak of, because nothing happens.

- The program does not read instruction data, so malformed or
  oversized data is tolerated silently (discarded).
- The program does not read accounts, so any accounts the caller
  passes will be ignored. (The runtime still loads them and charges
  for them in compute, so don't pass unnecessary accounts in
  practice.)
- The two `msg!` / `log!` lines cost compute units. With the default
  200 000 CU budget, they are a rounding error.

## 7. Running the tests

The four implementations each have their own test harness; all four
do the same thing: build the program, send a `hello` transaction,
assert the log contains "Hello, Solana!".

```bash
# Anchor
cd anchor && anchor build && anchor test

# Native (solana-program + LiteSVM / TypeScript)
cd native && cargo build-sbf && pnpm install && pnpm test

# Pinocchio
cd pinocchio && cargo build-sbf && cargo test

# Quasar
cd quasar && cargo test
```

The repo's `.github/workflows/anchor.yml`, `native.yml`,
`pinocchio.yml` and `quasar.yml` run the same commands in CI for
every changed project.

## 8. Extending the program

A few useful next steps:

- **Accept instruction data.** Read one byte off the instruction data
  and log it differently. Introduces `Borsh`-style deserialisation.
- **Take one account and read it.** Makes you learn how
  `AccountInfo` is arranged (owner, data, lamports) — the next
  example up, `basics/account-data`, does this.
- **Write to an account.** Now you're touching rent and signer checks
  — see `basics/create-account` and `basics/checking-accounts`.

## Framework differences

All four implementations are under ~25 lines and do the same thing.
The differences are in boilerplate and which crate holds the logging
primitive.

| Framework | Entrypoint | Logging | Noteworthy |
|---|---|---|---|
| `anchor/` | `#[program] mod ... { pub fn hello(...) }` | `msg!` | Generates an 8-byte discriminator for the `hello` method automatically |
| `native/` | `entrypoint!(process_instruction)` free function | `msg!` | No framework: just `solana-program` types directly |
| `pinocchio/` | `program_entrypoint!` macro + `no_allocator!` + `default_panic_handler!` | `log!` (from `pinocchio_log`) | Allocator-free runtime; compiles smaller than Anchor for equivalent work |
| `quasar/` | `#[program] mod ... { #[instruction(discriminator = 0)] pub fn hello(...) }` | Anchor-compatible via quasar-lang | `#![cfg_attr(not(test), no_std)]` — no std outside tests; minimal footprint |

Pick `native` if you want to see every byte, `anchor` if you want
generated boilerplate and IDLs, `pinocchio` if you're cost-sensitive,
`quasar` if you like Anchor's ergonomics without Anchor's runtime
size.
